宣言的 UI は「状態を渡せば表示が決まる」という考え方です。React を使っていれば `UI = f(state)` という言い方で一度は目にすると思います。ただ、実際に画面を組み始めると、どこまでを状態として持ち、どこからを描画側の責務にするかで手が止まる場面が出てきます。この記事では、そのあたりで判断が要るところを整理しておきます。

対象にしているのは、フレームワークの入門を終えて、実際のアプリで状態設計に迷い始めた人です。特定のライブラリの API そのものより、状態と描画の境界をどこに引くかに寄せて書きます。

> 状態を増やすほど画面は予測しやすくなりますが、同期すべき値も増えます。宣言的にするというのは、状態を最小限にして、残りを計算で導く作業でもあります。

:::banner{type="note"}

ここでいう宣言的 UI は、[React 公式ドキュメント](https://react.dev/learn/reacting-to-input-with-state) が説明している「状態に応じて UI を記述し、遷移そのものは書かない」スタイルを指しています。命令的に DOM を触る書き方と対比すると分かりやすいです。

:::

## まず状態にするものを絞る

画面の要素をすべて状態にすると、更新のたびに整合性を保つ責任が増えます。最初に決めたいのは、何を「持つ」かです。持つと決めるのは、次のような値です。

- ユーザーの入力そのもの（フォームの値、選択中のタブ）
- サーバーから取得したデータ
- 非同期処理の進行状況（`loading` や `error`）

### 派生値は状態にしない

逆に、次のようなものは状態にしないほうが扱いやすいです。

- 他の状態から計算できる値（合計金額、フィルタ後の一覧）
    - 派生値は描画時に求めれば、同期のずれが起きません
    - どうしても重いときだけ、メモ化を検討します
- DOM から読み取れる値（スクロール位置のように、参照で足りるもの）

作業としては、次の順番で見ていくと迷いにくいです。

1. 画面に必要な値を書き出す
2. そのうち、他の値から導けるものに印をつける
3. 印のついたものを状態から外し、計算に置き換える

## 状態から描画を書く

宣言的に書くと、コンポーネントは「今の状態ならこう見える」を返すだけになります。遷移のアニメーションや DOM 操作を手で書かない、というのがポイントです。

```tsx
type Status = "idle" | "loading" | "loaded" | "error";

function SubmitButton({ status }: { status: Status }) {
	const label =
		status === "loading" ? "送信中…" : status === "error" ? "やり直す" : "送信";

	return (
		<button type="submit" disabled={status === "loading"}>
			{label}
		</button>
	);
}
```

`status` という **一つの値** から、ラベルと `disabled` の両方が決まります。取り得る状態は次の四つに絞りました。

| 状態 | 意味 |
| --- | --- |
| `idle` | 何もしていない初期状態 |
| `loading` | 送信中で操作を止める |
| `loaded` | 完了して結果を出す |
| `error` | 失敗してやり直しを促す |

表示の分岐を状態に集約しておくと、あとで `error` の見た目を変えたいときも、この関数だけを読めば済みます。

## 命令的な書き方と比べる

同じ「送信ボタン」を、命令的に組む場合と宣言的に組む場合で並べてみます。列で比べると、どこに判断が移るかが見えます。

| 観点 | 命令的な実装 | 宣言的な実装 | 主な責務の置き場所 | つまずきやすい点 | この記事での立場 |
| --- | --- | --- | --- | --- | --- |
| 状態の保持 | DOM とローカル変数に散る | 一つの `status` に集約 | コンポーネントの state | 二重管理でずれる | 集約に寄せる |
| 表示の更新 | イベントごとに手で書き換え | 再描画に任せる | フレームワーク | 更新漏れ | 任せる |
| 分岐の追加 | 各所に `if` が増える | 分岐関数を一つ直す | 描画ロジック | 変更箇所が読めない | 一箇所にまとめる |
| テスト | 操作手順を再現する | 状態を渡して出力を見る | 入出力 | 手順依存で壊れやすい | 入出力で見る |

命令的な書き方が悪いわけではありません。フォーカス制御や要素の測定など、~~状態にしづらい~~ 参照で扱うべき処理は、命令的なほうが素直です。分けたいのは、状態から決まるものと、 *参照で扱うもの* を切り分けることです。

:::banner{type="warning"}

すべてを状態にすると、今度は状態同士の同期が新しいバグの温床になります。「派生できる値は状態にしない」を守らないと、宣言的にしたつもりで手動同期が増えるので注意します。

:::

## 状態遷移は図で持っておく

状態が三つを超えたら、遷移を一度図にしておくと実装が楽になります。取り得る状態と、そのあいだの矢印だけの簡単なもので十分です。

![media:019d1223-94d4-754c-8f57-47337be15c9e]()

図にすると、`loading` から `error` に戻る経路を書き忘れていた、といった抜けに気づけます。実装前に一度眺めておくと、`status` の型に入れ忘れがなくなります。

## 最後に確認すること

実装したあとで、次を見直しています。

1. 状態にした値は、本当に他から導けないか
2. 一つの状態から、複数の表示が決まっているか
3. 命令的に書いた箇所は、参照で扱うべき処理に限られているか

---

宣言的 UI は書き方の名前というより、状態を絞った設計の結果だと考えています。持つ値を減らせたぶんだけ、画面は読みやすくなります。
